Section 01
NPU vs TPU vs VPU — For Camera AI

These three processor types all run neural networks, but they're fundamentally different in design, purpose, power budget, and where they sit in your camera architecture. Understanding each one tells you which chip should live in your edge device at each workstation.

NPU
Neural Processing Unit
EDGE NATIVE
Purpose-built chip for running neural network inference at extremely low power. Executes matrix multiply–accumulate (MAC) operations in dedicated hardware circuits — the core operation of every neural network layer. Found inside: Jetson Orin (NVDLA), Apple Neural Engine, Hailo-8, Rockchip RK3588, Kneron KL720, Ambarella CV72.
Primary JobInference (running models)
Power Draw0.5W – 5W typical
Can Train?NO — inference only
PrecisionINT4, INT8, FP16
TOPS5 – 26 TOPS (edge)
For Your Project★ Primary inference chip
Your Use: The NPU inside Jetson Orin Nano/NX runs your YOLOv8 model at 60fps using only 5–15W per station. This is the real-time inference engine that powers all 50 cameras. Training never happens on the NPU — it's a runner, not a learner.
TPU
Tensor Processing Unit
TRAIN + INFER
Google's custom ASIC, designed specifically for TensorFlow operations. Unlike NPUs which focus on inference, TPUs are built for both training AND inference at scale. The Edge TPU (Google Coral) is a compact version at 4 TOPS for 2W. Full Cloud TPUs (v4, v5p) are enormous data-centre chips used by Google internally to train models like Gemini.

Can Train?Edge TPU: NO / Cloud: YES
PrecisionINT8 (Edge), BF16 (Cloud)
TOPS4 TOPS (Coral) / Massive (Cloud)
For Your ProjectOptional: Cloud TPU for training
Your Use: Google Coral USB accelerator (₹3,500) can boost inference on a Raspberry Pi — adds 4 TOPS for USB-C. Cloud TPU (via Google Colab or GCP) can be used for large-scale teacher model training. Edge TPU is TensorFlow-only which limits YOLOv8 compatibility.
VPU
Vision Processing Unit
VISION SPECIALIST
Designed specifically for vision workloads: ISP (image signal processing), feature extraction, optical flow, depth estimation, object tracking. Intel Myriad X is the landmark VPU — 4 TOPS, 1W, on-board 16-lane stereo vision accelerator. Used in Intel RealSense depth cameras, OpenVINO ecosystem. Hailo-8 is a modern "VPU-NPU hybrid" at 26 TOPS specifically for vision inference.

🔑 The Core Insight on NPU/TPU/VPU
None of these chips can train a model. They are all inference-only processors. The word "self-training" in camera systems always means: the camera collects data (using NPU/VPU for inference), the data goes to a central GPU server for training, and the new model comes back to the camera. The NPU's job is to run that returned model as fast and efficiently as possible. In your architecture: Jetson Orin Nano (NPU + GPU for inference on-device) + Central RTX 4060 server (GPU for training) = complete teacher-student loop.
Section 02
Teacher → Student Learning Architecture

Teacher-Student (Knowledge Distillation) is the technique of compressing a large, accurate "teacher" model into a small, fast "student" model — while preserving most of the teacher's accuracy. For camera systems, this is how you get a model smart enough to run at 60fps on a 5W NPU.

🎓 Why Teacher-Student Matters for Your Assembly Line
Your Teacher model (YOLOv8x, runs on RTX 4060, 50ms inference, high accuracy) is what you use to label new data, generate pseudo-labels, and validate student outputs. Your Student model (YOLOv8n, quantized INT8, runs on Jetson NPU, 3ms inference, 5W) is what's actually deployed at all 50 workstations. The teacher never leaves the server. The student lives in the camera.
🎓

Stage 1 — Teacher Learns First (Fully Supervised)
At V1 deployment, a large teacher model (YOLOv8m or YOLOv8x) is trained on your manually labeled factory dataset. This takes 2–6 hours on an RTX 4060. The teacher is heavy (50–70MB, 50ms/frame) but highly accurate (>90%). It stays on the central server and is never deployed to cameras. Its job is to be the authority — the ground truth generator for all future training.
# Teacher training — runs once on central GPU server from ultralytics import YOLO teacher = YOLO('yolov8m.pt') # Medium model = teacher teacher.train( data='factory_dataset.yaml', epochs=150, imgsz=640, batch=16, device='0' # RTX 4060 ) # ~3–5 hours on RTX 4060

Student
Model
Stage 2 — Distill Teacher → Student
The student model (YOLOv8n — nano) is trained to mimic the teacher's soft output probabilities — not just the hard labels. Instead of learning "this is Bolt-A (class 1)", the student learns "the teacher is 89% sure this is Bolt-A, 7% Bolt-B, 4% Washer-C." These probability distributions carry richer information than binary labels. The student learns from how the teacher distributes confidence across all classes, capturing nuanced relationships even when predictions are incorrect. This is called "soft-target" or "dark knowledge" learning.
# Knowledge distillation: student learns teacher's soft outputs # Temperature T > 1 softens probabilities, exposing more relationships import torch.nn.functional as F # Teacher output (soft) — at temperature T=4 teacher_soft = F.softmax(teacher_logits / 4.0, dim=-1) # Distillation loss: student matches teacher's soft distribution dist_loss = F.kl_div( F.log_softmax(student_logits / 4.0, dim=-1), teacher_soft, reduction='batchmean' ) * (4.0 ** 2) # scale by T² # Total loss = 0.3 × hard label loss + 0.7 × distillation loss total_loss = 0.3 * ce_loss + 0.7 * dist_loss

Stage 3 — Quantise & Deploy Student to NPU
The distilled student model (YOLOv8n, ~6MB, FP32) is then quantised from 32-bit float to INT8 — halving inference time and memory. The INT8 model is compiled via TensorRT into a .engine file optimised for the Jetson NPU. Despite a slight drop in accuracy, this achieves remarkable reductions in memory usage and inference time, making it a viable solution for real-time deployment on resource-constrained NPUs. The engine file is pushed OTA to all 50 stations.
# Export student to TensorRT INT8 for Jetson NPU student = YOLO('student_yolov8n_distilled.pt') student.export( format='engine', # TensorRT int8=True, # INT8 quantisation device='0', imgsz=640 ) # Result: student_yolov8n_distilled.engine (~3MB, 3ms/frame) # vs Teacher: yolov8m.pt (~50MB, 50ms/frame) # Accuracy trade-off: ~2-3% mAP drop, 16× speed gain

Stage 4 — The Continuous Teacher-Student Loop (V2)
After deployment, the teacher continues working. When new data arrives (new product, new SOP), the teacher auto-labels it (generates pseudo-labels). The student is re-distilled on teacher pseudo-labels + original labeled data. A new, smarter student gets deployed. The teacher never needs to be deployed — it gets better over time on the server. The student stays light, fast, and gets smarter versions pushed to it.

Section 03
Training Without Annotation — The Logic Explained

In V2, when a new product arrives or SOP changes, you shouldn't need to manually label thousands of images every time. There are 3 techniques that enable zero or minimal annotation training — here's the exact logic behind each one.

Technique 01
Pseudo-Labeling
The teacher model runs on unlabeled new images. Any detection with confidence > 70% becomes a "pseudo-label" — treated as if it were manually annotated. The student retrains on these auto-generated labels. Only detections the teacher is confident about are trusted. Uncertain ones are discarded or sent for human review.
Technique 02
Semi-Supervised Learning
A small set of labeled images (e.g. 100 manually labeled) is used alongside a large pool of unlabeled images. The model trains on labeled data first, then uses its learned knowledge to generate pseudo-labels for unlabeled data, which it also trains on. This "bootstrapping" expands the effective dataset from 100 to 10,000+ usable examples.
Technique 03
Mean Teacher (Self-Training)
Two identical models run simultaneously. The "mean teacher" is an exponential moving average (EMA) of the student's weights — it's always slightly ahead. The teacher generates pseudo-labels on unlabeled data. The student trains on them. Then teacher's weights are updated by averaging with the student's latest weights. They pull each other forward. No human labels needed.
Technique 04
Consistency Regularization
The same image is fed to the model twice — once normally, once with augmentation (brightness, rotation, noise). Both outputs should predict the same objects. If they don't, the model is penalised. This trains the model to be robust without needing labels — just by enforcing internal consistency on unlabeled data.
Technique 05
Active Learning Loop
The model flags frames where it's most uncertain (confidence 40–65%). These are the frames where a human label gives the most learning value. A human reviews only these ~200 frames out of 10,000 captured. This 98% annotation reduction while maintaining near-full accuracy improvement is the most practical technique for your factory V2 pipeline.
Technique 06
Contrastive Self-Supervised Learning
The model learns visual representations by contrasting similar and dissimilar frames — without any labels. "These two frames both show a partially assembled engine block, so they should have similar feature vectors. This frame shows a bare frame — it should be different." The model learns what objects look like before it knows what to call them. Features learned here transfer to detection with few labels.

🧠 The Key Threshold: 70% Confidence Rule
The 70% confidence threshold is the most important tuning parameter in your pseudo-label pipeline. Too high (90%+): very few pseudo-labels generated, not enough data to retrain on. Too low (50%-): noisy pseudo-labels, model trains on wrong examples and gets worse. 70–75% is the empirically proven sweet spot for factory vision tasks. This threshold should be configurable in your dashboard, not hardcoded.

Full training from scratch (on COCO: 80 classes, 118K images): 100–300 hours on a single GPU. Transfer learning fine-tuning (your scenario: 5–15 classes, 500–3000 images, start from pretrained weights): 15 minutes to 3 hours on an RTX 4060. You almost never train from scratch — you always fine-tune. The pretrained weights are your starting point; fine-tuning adjusts only the last few layers for your specific factory data.

Section 05
User-Triggered SOP / Object / Action Update Loop

When the user wants to update an SOP, add a new object class, or teach the camera a new worker action — this is the exact end-to-end flow from user input to updated camera model. This is the heart of V2.

What triggers a model update?
  • New product model introduced — new parts never seen by current model
  • SOP sequence changed — steps reordered or new tool added
  • New action added — new worker gesture or tool movement to detect
  • Part supplier changed — same part, different visual appearance
  • Confidence drift alert — system auto-detects model getting confused
  • Manual refresh request — supervisor decides to improve a station's model
What the user sees (UI actions)
  • Go to Dashboard → Station Settings → Model Update
  • Choose update type: New Object / New Action / SOP Rule Change
  • Upload: video clip OR folder of images
  • Add class names (text only: "bolt_m8_zinc" or "torque_wrench_action")
  • Click "Start Training" — everything else is automatic
  • Review metrics when ready, click "Deploy to Station"
  • Optional: deploy to just 1 station first to test, then roll to all
The 8-Step Model Update Loop — Complete Flow
USER
Phase 1 — USER ACTION
Upload + Describe
User uploads a short video (2–5 min recording of the new part/action) or a folder of 50–200 photos to the web portal. They type the class name(s) for new objects/actions in a simple form. If it's an SOP rule change (no new objects), they use the SOP Config Editor — no video upload needed. The portal accepts: MP4, MOV, AVI, JPG, PNG.
Web Portal Video / Images Class Names Station Selection
AUTO
Phase 2 — EXTRACTION
Frame Extraction + Deduplication
Server receives the upload. A Python job (Celery worker) extracts frames from video at 1 frame every 10 frames (sampling rate configurable). Duplicate or near-duplicate frames are removed using perceptual hashing (pHash). Result: typically 200–600 unique, diverse frames from a 3-minute video. These are staged in the "pending label" folder.

AUTO
Phase 3 — TEACHER AUTO-LABEL
Pseudo-Label Generation + Confidence Split
The teacher model (YOLOv8m on the server) runs inference on all 400 frames. Each detected object gets a bounding box, class label, and confidence score. High-confidence frames (≥70%): auto-approved as pseudo-labeled training data. Low-confidence frames (40–70%): flagged for human review queue. Very low (<40%): discarded. For new class names that the teacher has never seen, ALL frames go to human review.
Auto-label: ~250 frames Human queue: ~100 frames Discarded: ~50 frames
USER
Phase 4 — HUMAN REVIEW (Optional but Recommended)
Label Correction in Review Portal
The dashboard shows the ~100 uncertain frames in a label review UI (similar to Roboflow, built into your dashboard). Pre-drawn bounding boxes are shown — user clicks "Accept" or "Correct" on each. For totally new objects, user draws boxes on the ~100 frames manually. This typically takes 20–45 minutes for one trained reviewer. If skipped entirely, the model trains on pseudo-labels only — accuracy will be slightly lower but still improves.

AUTO
Phase 5 — DATASET MERGE & FINE-TUNE
Retrain Student on Teacher Labels + Original Data
Pseudo-labels + reviewed labels are merged with the master training dataset. The existing student model is fine-tuned (not trained from scratch — only the last few layers update, preserving knowledge of old classes). Fine-tuning runs for 30–50 epochs. With 500–1000 total images and an RTX 4060, this takes 12–28 minutes. The learning rate is set low (0.001) to avoid catastrophic forgetting of old knowledge.

AUTO
Phase 6 — EVALUATION GATE
Automated Benchmark — Must Beat Old Model
The new student model is evaluated on the golden test set (100 manually labeled images saved at V1 deployment and never used for training). Three gates must pass: (1) New model mAP50 ≥ old model mAP50 - 0.02 (allows max 2% degradation on old classes). (2) New class detection rate ≥ 75%. (3) False positive rate on known-good assemblies ≤ 5%. If any gate fails, the model is rejected and the system notifies the user with the exact reason.
Gate 1: Old class accuracy Gate 2: New class accuracy Gate 3: False positive rate
USER
Phase 7 — SUPERVISOR APPROVAL
1-Click Deploy Approval
The dashboard sends a notification to the designated supervisor. They see: model accuracy report, comparison to previous model, which stations will be updated, and the option to deploy to 1 station first (canary test) or all affected stations. One click on "Deploy" starts the OTA push. No code required. The supervisor doesn't need to understand ML — just review: "Is new accuracy higher? Yes? Deploy."
AUTO
Phase 8 — OTA DEPLOYMENT
Model Pushed to Cameras — Zero Downtime
The new .engine file (INT8 quantised TensorRT model, ~3–5MB) is pushed from the central server to the selected edge devices over the factory LAN. Each edge device: downloads to /models/staging/, verifies MD5 checksum, waits for the current station to enter IDLE or HANDOFF mode, then atomically swaps the active model file. Old model moves to /models/backup/. If the new model crashes or causes error spike within 30 minutes, auto-rollback triggers. The camera is smarter — users see no interruption.
3–5MB file push Checksum verified Zero downtime swap Auto-rollback guard
⚠️ Critical Design Rule: SOP Config ≠ Model Update
When the user changes the SOP rule (e.g. "Step 4 now requires Torque Wrench-B instead of Wrench-A"), there are TWO independent systems to update:

1. SOP Config file (YAML) — update this immediately via the SOP Config Editor. No training needed. Takes 2 minutes. The rule engine reads this file.

2. Detection model — update this only if the camera needs to recognise a NEW visual object (Torque Wrench-B was never in training data). If the object is already known to the model, only the config changes — model stays same.

Most SOP changes don't require model retraining — only config changes. This distinction saves enormous time.
